NAME
tail â copy the last part of a file

SYNOPSIS
tail [âf] [âc number|ân number] [file]

DESCRIPTION
The tail utility shall copy its input file to the standard output beginning at a designated place.

Copying shall begin at the point in the file indicated by the âc number or ân number options. The option-argument number shall be counted in units of lines or bytes, according to
the options ân and âc. Both line and byte counts start from 1.

Tails relative to the end of the file may be saved in an internal buffer, and thus may be limited in length. Such a buffer, if any, shall be no smaller than {LINE_MAX}*10 bytes.

The origin for counting shall be 1; that is, âc +1 represents the first byte of the file, âc â1 the last.

OPERANDS
The following operand shall be supported:

file A pathname of an input file. If no file operand is specified, the standard input shall be used.

STDIN
The standard input shall be used if no file operand is specified, and shall be used if the file operand is ’â’ and the implementation treats the ’â’ as meaning standard input.
Otherwise, the standard input shall not be used. See the INPUT FILES section.

INPUT FILES
If the âc option is specified, the input file can contain arbitrary data; otherwise, the input file shall be a text file.

APPLICATION USAGE
The âc option should be used with caution when the input is a text file containing multi-byte characters; it may produce output that does not start on a character boundary.

Although the input file to tail can be any type, the results might not be what would be expected on some character special device files or on file types not described by the Sysâ
tem Interfaces volume of POSIX.1â2008. Since this volume of POSIX.1â2008 does not specify the block size used when doing input, tail need not read all of the data from devices
that only perform block transfers.

EXAMPLES
The âf option can be used to monitor the growth of a file that is being written by some other process. For example, the command:

tail âf fred

prints the last ten lines of the file fred, followed by any lines that are appended to fred between the time tail is initiated and killed. As another example, the command:

tail âf âc 15 fred

prints the last 15 bytes of the file fred, followed by any bytes that are appended to fred between the time tail is initiated and killed.

RATIONALE
This version of tail was created to allow conformance to the Utility Syntax Guidelines. The historical âb option was omitted because of the general non-portability of block-sized
units of text. The âc option historically meant ‘‘characters’’, but this volume of POSIX.1â2008 indicates that it means ‘‘bytes’’. This was selected to allow reasonable implemenâ
tations when multi-byte characters are possible; it was not named âb to avoid confusion with the historical âb.

The origin of counting both lines and bytes is 1, matching all widespread historical implementations. Hence tail ân +0 is not conforming usage because it attempts to output line
zero; but note that tail ân 0 does conform, and outputs nothing.

Earlier versions of this standard allowed the following forms in the SYNOPSIS:

tail â[number][b|c|l][f] [file]
tail +[number][b|c|l][f] [file]

These forms are no longer specified by POSIX.1â2008, but may be present in some implementations.

The restriction on the internal buffer is a compromise between the historical System V implementation of 4096 bytes and the BSD 32768 bytes.

The âf option has been implemented as a loop that sleeps for 1 second and copies any bytes that are available. This is sufficient, but if more efficient methods of determining
when new data are available are developed, implementations are encouraged to use them.

Historical documentation indicates that tail ignores the âf option if the input file is a pipe (pipe and FIFO on systems that support FIFOs). On BSD-based systems, this has been
true; on System V-based systems, this was true when input was taken from standard input, but it did not ignore the âf flag if a FIFO was named as the file operand. Since the âf
option is not useful on pipes and all historical implementations ignore âf if no file operand is specified and standard input is a pipe, this volume of POSIX.1â2008 requires this
behavior. However, since the âf option is useful on a FIFO, this volume of POSIX.1â2008 also requires that if a FIFO is named, the âf option shall not be ignored. Earlier verâ
sions of this standard did not state any requirement for the case where no file operand is specified and standard input is a FIFO. The standard has been updated to reflect curâ
rent practice which is to treat this case the same as a pipe on standard input. Although historical behavior does not ignore the âf option for other file types, this is unspeciâ
fied so that implementations are allowed to ignore the âf option if it is known that the file cannot be extended.
